Threat Modeling
Assistant

Une plateforme open-source qui automatise la génération de modèles de menaces STRIDE et LINDDUN pour les applications Android — en combinant l'analyse statique d'APK, l'élicitation de contexte assistée par LLM et la correspondance avec les contrôles OWASP MASVS v2.

Motivation

Le problème

La modélisation des menaces est reconnue par l'OWASP, le NIST SP 800-154 et l'ISO/IEC 27034 comme une activité fondamentale du développement sécurisé — pourtant son adoption dans les équipes mobiles reste limitée. Produire un modèle manuellement prend 2 à 5 jours, nécessite une expertise pointue, et les outils existants ne proposent aucune analyse spécifique à Android.

Pour les applications Android, la surface d'attaque est encore plus complexe du fait du modèle de composants (activités, services, broadcast receivers, content providers) et de la nécessité de mapper chaque menace aux contrôles OWASP MASVS v2 répartis en 7 familles.

TMA comble ce manque en combinant l'analyse statique d'APK via Androguard, l'élicitation de contexte par LLM et la génération structurée STRIDE / LINDDUN — le seul outil open-source à réunir simultanément ces quatre capacités.

Fonctionnalités

Ce que fait TMA

🔍

Analyse statique d'APK

Extrait les permissions, les composants exportés et les endpoints API depuis le binaire APK via Androguard — sans code source requis.

🧠

Élicitation de contexte par LLM

Génère 6 questions ciblées à partir du contexte extrait avant de synthétiser le modèle de menaces. Compatible Anthropic, OpenAI, Ollama ou mode mock.

⚔️

STRIDE & LINDDUN

Supporte les deux principaux frameworks de threat modeling — STRIDE pour la sécurité et LINDDUN pour la vie privée — sélectionnables par session.

🛡️

Mapping OWASP MASVS v2

Chaque menace est automatiquement mappée aux contrôles MASVS v2 sur les 7 catégories : STORAGE, CRYPTO, AUTH, NETWORK, PLATFORM, CODE, RESILIENCE.

🌐

Bilingue FR / EN

Interface complète et résultats générés en français et en anglais. La préférence de langue est persistante entre les sessions.

📦

Déploiement en une commande

Entièrement conteneurisé avec Docker Compose. Le mode mock ne nécessite aucune clé API — zéro configuration pour l'exploration et la CI.

Architecture

Comment ça fonctionne

TMA suit une architecture client-serveur à trois niveaux. nginx route les requêtes /api/ vers le backend FastAPI tout en servant le SPA React à la racine.

Frontend — React 18 + Vite · port 3000
UploadZone QuestionForm ThreatModelView RiskRegister MitigationPanel i18n (FR/EN)
↕ HTTP/JSON · proxy inverse nginx
Backend — FastAPI · Python 3.12 · port 8000
POST /api/analyze POST /session/{id}/generate GET /api/export/{id} apk_extractor threat_modeler Session store (TTL 1h)
↕ LLMProvider ABC
Couche d'abstraction LLM
mock (défaut) claude_code anthropic openai ollama openai_compatible

Workflow en 3 étapes

01 / UPLOAD

APK + Description

Déposez un APK (jusqu'à 100 Mo) et fournissez une description textuelle de l'application.

02 / QUESTIONS

Élicitation de contexte

Répondez aux 6 questions générées par le LLM. Sélectionnez le framework STRIDE ou LINDDUN avant de lancer la génération.

03 / RÉSULTATS

Modèle de menaces

Onglets Menaces · Registre des risques · Mitigations MASVS. Export du modèle complet en JSON.

Démarrage rapide

# Cloner et lancer — aucune clé API requise (mode mock) git clone https://github.com/Eti-Hack/Threat_Modeling_Assistant.git cd Threat_Modeling_Assistant cp .env.example .env docker compose up --build # Ouvrir http://localhost:3000 # Pour utiliser un vrai LLM : echo "LLM_PROVIDER=anthropic" >> .env echo "ANTHROPIC_API_KEY=sk-ant-..." >> .env docker compose restart backend
